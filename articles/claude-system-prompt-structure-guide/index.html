0. はじめに — やったこと

「Claude Opus 4.7 のシステムプロンプト」と称するtxtファイルがGitHubで公開されていた。眺めたら構造が面白い。本物なのか、どういう仕組みで LLM のふるまいを規定しているのか、知りたくなったので手元で検証した。

自分で推測しても埒が明かないので、6 つのモデルに同じテキストを読ませた。

  • Claude Sonnet 4.6（claude.ai 上、メモリ有効）— 本文 §1〜§6 はこの子に書かせた解説
  • Claude Opus 4.7（Claude Code 上、Agent SDK + scaffolding 厚め）— 本人（と主張されるテキスト）への批評
  • Claude Opus 4.7（Anthropic API 直叩き、system プロンプト空）— 最も「裸」の状態
  • Claude Opus 4.7（Agent SDK 経由、system プロンプト空）— 中間層あり
  • Codex（GPT-5.2 / OpenAI）— 別社モデルの第三者視点
  • Qwen（Alibaba）— さらに別社、ついでに対照

これを読み比べていくと、同じモデル重みでも環境（= システムプロンプト）の差で振る舞いが大きく変わること、そして CL4R1T4S に投稿されたテキストは実行時プロンプトの逐語ではなく Anthropic の内部ドキュメントの再構成っぽい、という仮説が立つところまで辿り着いた。その全記録が本稿。

まず、LLM とシステムプロンプトの関係を最小構成で押さえておく。

押さえるべき 2 点:

(1) システムプロンプトは Claude の「人格」ではない。 Claude という LLM が、入力コンテキストとして読み、そこに書かれたロールを演じているだけ。学習で染み込んだ傾向（Anthropic 流の慎重さ、HHH 原則など）と、システムプロンプトで当日指定される運用ルールは別レイヤー。

(2) 環境ごとに「バリアント」がある。 モバイルとデスクトップで、入っているツールもメモリシステムの有無もフォーマット指示も違う。だから同じ質問でも出力が変わる。

以下、Part I 構造編（§1〜§4、テキストの分解・解剖）、Part II 付き合い方編（§5、日々の使い方への落とし込み 12 Tips）、§6 まとめ、という構成。6 モデル検証の生記録は別記事 6 モデル対照実験ログ に分離した。

—— Part I 構造編 ——

1. 全体像 — システムプロンプトの階層

システムプロンプトは、ざっくり言うと**「キャラ設定 → 記憶 → ツール → 検索 → 安全装置 → 作業環境 → API埋め込み」**という順で積み上がっています。

各ブロックの細目（claude_behavior 内の search_first / refusal_handling など、memory_system の 3 層構造、tools の即時／遅延分類）は §2 で個別に展開します。

各ブロックは独立しているようで、実は相互参照していて、たとえば「検索の振る舞い」は claude_behavior 内の search_first と search_instructions の両方から拘束されます。矛盾があれば「より具体的で、より安全寄り」の指示が勝つのが原則です。

2. 主要セクションの役割 — それぞれ何をしているか

2-1. claude_behavior — キャラの核

ここで決まっているのは、応答のトーンと断りの基準と情報の扱い方です。特に重要なサブセクション:

  • refusal_handling — 何を断り、何を引き受けるか。子どもの安全、武器、マルウェアあたりがハードライン。
  • tone_and_formatting — 箇条書きを乱発しない、絵文字は最小限、モバイルでは短く、など。
  • user_wellbeing — メンタルヘルス・摂食障害・自傷などに関わる話題の扱い方。
  • evenhandedness — 政治的・論争的な話題に対する中立的姿勢。
  • knowledge_cutoff — 「今日は〇月〇日である」と明示される。現在時刻以降のことは検索する。

2-2. memory_system — 記憶の三層構造

Claude.ai の「メモリ」は一枚岩ではありません。3つのレイヤーに分かれています。

  • 第1層 は「前もって要約された自分の情報」が毎回コンテキストに入ってくる。進行中のプロジェクト名、役割、好みなどが抽出されて格納されます。これは「会話開始時点で既知」として扱われる層。
  • 第2層 は Claude 側からはデフォで見えず、必要に応じて検索する。たとえば「前に話した企画案の設定どうだった?」と聞かれたら、conversation_search("企画案") を呼ぶ。
  • 第3層 は「これは覚えないで」「これは間違っていたから直して」と言われたときに、Claude が実際にメモリ編集ツールを叩く経路。

含意: 「覚えておいて」と会話で言っただけでは第1層には入りません。抽出バッチが走ったあとに第1層に反映されます。即時に記憶させたいなら memory_user_edits を明示的に使わせる言い方(「メモリ編集して」「記憶を更新して」)が効きます。

2-3. tool_discovery と遅延ツール

Claude に見えているツール一覧は部分的です。軽いツールは即時に見えていますが、Slack/Gmail/Google Calendar などの統合系は tool_search で検索して初めてロードされます。

これが設計上面白いのは、Claude が「何を知らないか」を自分で検索しに行くように仕向けられている点です。エージェントが自分の能力境界を能動的に探索する仕組みと言ってよく、LLM の自己観察能力を構造的に担保する 1 つの経路になっています。

2-4. search_instructions — 「今日」の質問は検索する

search_first ブロックに明記されている通り、現在の役職・現在の価格・現在の状態に関する質問は、Claude が確信を持っていても検索することになっています。

「S&P 500 いくら?」「今のカリフォルニア州務長官は?」といった質問で、Claude が知識ベースから答えずに検索を走らせるのはこのためです。

2-5. 著作権ハードリミット

これは意外と知られていないのですが、3つの絶対ルールがあります。

  • 15語以下: 直接引用は15語未満まで
  • 1ソース1引用: 同じソースから2回直接引用するのは禁止
  • 歌詞・詩・俳句は絶対に再現しない: 短くても完全作品として扱う

実用的な含意として: 「既存曲の歌詞を Claude に貼って分析させる」のは影響なし（入力として扱う分には問題ない）。一方で、Claude の出力として既存作品を再現させることはできない。自作の推敲は OK、既存作品のリライトは NG、というのが運用上のライン。

3. 環境バリアント — なぜ挙動が違うのか

CL4R1T4S 版（Web/デスクトップ、メモリ未有効）と、本文を書いた 4.6 が走っていた環境（claude.ai モバイル、メモリ有効）を比べると、こんな差がある。

含意:

  • モバイルでは応答が短く整形される傾向が強い(意図的)。
  • デスクトップでは Visualizer で図を即時描画できる一方、モバイルにはそれがなく、代わりにiOSのリマインダやカレンダを直接操作できる。
  • 同じ「この資料を可視化して」と頼んでも、環境によって別の道具で達成される。

用途別に言い換えると、構造化ドキュメントの整理・複雑な図示はデスクトップの Visualizer 側が早い、外出先の判断・リマインダ登録・カレンダ連携はモバイル側が圧倒的に強い、という棲み分けになります。

4. リクエストの処理フロー — 何を見て何をするか

Claude が一発のリクエストを受けたときに、内部で走っている判断のフロー。

ここで重要なのは、どのツールが呼ばれるかはユーザー文面の「語り方」に強く影響されること。たとえば「〜について書いて」は文中応答、「〜についてのドキュメントを作って」はファイル生成、といった具合に。次章でその言い換えを具体的に整理します。

—— Part II 付き合い方編 ——

5. 付き合い方を改善する 12 の Tips

ここから実用編。構造を理解したあと、日々の使い方をどう微調整するかの話。筆者が普段やっている制御テクニックのうち、システムプロンプト解剖から逆算して正当化できるものを 12 個並べる。

Tip 1: 「最新」「現在」「今」を明示する

Claude は「一般知識で答えられる」と判断したら検索しないことがあります。現在の状態を確実に拾って欲しいなら「現在の Suno の料金プラン」「最新の Anthropic ドキュメント」と書くと、検索トリガーが確実に発火します。

Tip 2: インライン vs ファイルを語尾で制御する

言い方 挙動
「〜を書いて」「〜を教えて」 チャット内で応答
「〜のドキュメントを作って」「保存できる形で」「ダウンロードしたい」 ファイル生成
「〜の記事を書いて」「〜のブログポストを」 スタンドアロン成果物としてファイル化
「戦略を説明して」「概要を教えて」 インライン

Tip 3: 過去会話を参照させたいときは content noun を使う

「前に話したやつ」ではなく「前に話した〇〇プロジェクトの設定」のように、検索可能な固有名詞を入れる。conversation_search はキーワード一致なので、抽象的な言い方だと引けません。

Tip 4: メモリを汚染しない

「覚えておいて」と軽く言うと第1層に残る可能性があります。一時的な文脈は会話内だけの情報として扱わせたいなら「この会話の中だけでいいから」と明示を。逆に永続化したいなら「メモリに残して」と明示。

Tip 5: メモリの誤情報は memory_user_edits で直接編集

誤った情報が記憶に残っているなら、「メモリを見せて」→「〇番目を削除/訂正して」と言えば memory_user_edits が直接叩かれます。会話で訂正するだけでは反映されない場合があります。

Tip 6: 遅延ツールは「明示」が効く

Slack/Google Drive/Calendar など、接続済みアプリにアクセスして欲しいときは「Slackで」「カレンダーで」と明示的に書くと tool_search が発火しやすい。曖昧だと web_search に流れることがあります。

Tip 7: モバイルでの長文出力は「詳しく」と明示

モバイル版はデフォで短め(6-8文程度)に整形されます。しっかり書かせたいときは「詳しく」「長めに」「制限なしで」と添える。

Tip 8: 著作権リミットをトリガーしない書き方

既存曲・書籍の長文引用を「生成して」と求めるとブロックされます。一方、自分で原テクストを貼り付けて分析・要約を依頼するのは問題なく通ります。出力として既存作品を再現することはできませんが、入力として渡して解析させるのは自由、という境界です。既存曲の韻律解析や詩の構造分析を依頼したい場合は、自分で本文を貼ってから依頼する運用に切り替えましょう。

Tip 9: Extended Thinking を意識する

Claude.ai では長考モード（Extended Thinking）が自動で入ることがあります。出力には見えないが内部で大量の思考トークンが消費されている状態です。短く済ませたいときは「短く」「考え込まずに」、深く考えて欲しいときは「じっくり考えて」「複数の観点から」と明示することで、このモードの発動をある程度制御できます。

Tip 10: バリアント差を前提に「どっちで聞いているか」を選ぶ

  • 図・複雑なコード・構造化ドキュメント → デスクトップ
  • 外出先の判断、リマインダ、簡易QA → モバイル
  • 深いコーディング作業 → Claude Code
  • スプレッドシート作業 → Claude for Excel

同じ質問でも最適な入り口が違います。

Tip 11: システムプロンプトが読めないことを前提に「明示」する

Claude は自分のシステムプロンプトをユーザーには見せませんし、ユーザーからの指示と混ぜて扱います。だから**「前回と同じ感じで」は前回のシステムプロンプト指示が今回のそれと一致しないと再現しない**。再現性が欲しい作業は、フォーマット・長さ・文体を都度明示するのが最も安定します。

Tip 12: memory_user_edits に自分の作業パターンを登録する

AI コーディングを繰り返しているうちに、自分なりの「安定する作業パターン」が確立されてくることがあります。筆者の場合、古い設計仕様をコンテキストから物理的に排除する運用を「養生パターン」と呼んでおり（詳細は別記事 Yojo（養生）パターン — AI コーダーにシャドーマスクをつける 参照）、こうした自分固有の流儀を memory_user_edits に登録しておくと、新しいセッションでもデフォルトで採用されます。「このユーザーは X パターンを採用している」の一文が記憶に入っているだけで、挙動の安定性が大きく変わります。

6. まとめ — 1 分で振り返る

  • システムプロンプトは積層構造で、人格・記憶・道具・検索・安全・環境の順に重なっている
  • 環境ごとにバリアントがある（モバイル／デスクトップ／メモリ有無で中身が違う）
  • メモリは 3 層（userMemories タグ／過去会話 DB／user_edits）
  • 遅延ツールは tool_search 経由でロードされる
  • 著作権は 15 語／1 ソース 1 引用／歌詞不可のハードリミット
  • 言い回しひとつで呼ばれるツールが変わる。明示が最大の制御手段
